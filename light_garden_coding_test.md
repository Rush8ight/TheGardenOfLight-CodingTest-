# MSW 코딩테스트 '빛의정원'

**MSW 아르테일 클라이언트 개발자 코딩 테스트 Ver.1.1**  
**문서명:** 빛의정원  
**버전:** 1.1  
**문서 목적:** MSW 아르테일 클라이언트 개발자 채용용 코딩 테스트 안내서  
**테스트 유형:** Take-home 과제 / MOD 파일 제출 / 후속 코드 리뷰 인터뷰 포함  
**제출 원칙:** 기간 내 제출 필수. 미완성 상태라도 구현 범위, 미구현 범위를 함께 정리하여 제출하여 주시길 바랍니다.

---

## 1. 테스트 개요

본 테스트는 MapleStory Worlds 환경에서 MMORPG 라이브 서비스용 클라이언트 기능을 구현할 수 있는지 평가하기 위한 과제입니다.

지원자는 지금부터 **빛의정원 RPG**를 개발하는 클라이언트 개발자입니다.

빛의정원 RPG는 다음 특징을 가진 MMORPG입니다.

- 여러 플레이어가 동시에 접속할 수 있는 멀티플레이 RPG
- 몬스터 사냥, 경험치, 레벨업, 아이템 획득, 상점, 장비 교체가 포함된 성장형 게임
- 한국어와 영어를 지원하는 다국어 서비스
- 운영자 전용 커맨드와 운영 로그가 필요한 라이브 서비스
- 향후 거래 시스템 추가를 고려한 보안/데이터 무결성 설계가 필요한 서비스

본 테스트는 단순히 기능이 동작하는지만 평가하지 않습니다. 다음 역량을 종합적으로 평가합니다.

- MSW 환경에서의 기능 구현 능력
- 서버/클라이언트 실행 환경 이해도
- 멀티플레이 동기화 대응 능력
- 보안/악용 방지 설계 능력
- 데이터 저장/불러오기 안정성
- 운영자 기능 구현 및 권한 검증 능력
- 로컬라이징 구조 설계 능력
- 로그 및 라이브 운영 대응 능력
- AI 도구 활용 결과에 대한 검증 능력
- 코드 품질, 유지보수성, 확장성
- 제공된 레거시 구조와 리소스 사양을 읽고 요구사항에 맞게 확장하는 능력

본 과제의 모든 요구사항은 평가 대상입니다.
다만 구현 시간이 부족한 경우, 지원자는 미완성 범위와 사유를 명확히 제출해야 하며, 서버 검증, 데이터 저장, 전투/보상 중복 방지, 멀티플레이 안정성은 핵심 평가 항목으로 봅니다.

## 2. 테스트 기본 시나리오

유저는 게임 최초 접속 시 다음 상태로 시작합니다.

| 항목   | 초기값 |
| ------ | -----: |
| 레벨   |      1 |
| 경험치 |      0 |
| 메소   |    500 |

게임 진행 흐름은 다음과 같습니다.

1. 유저는 최초 마을 **오르비스**에서 시작합니다.
2. NPC **요정 에델**에게서 **우드 완드**를 구매 후 장착합니다.
3. 오르비스 오른쪽 포탈을 통해 **빛의정원** 맵으로 이동합니다.
4. 빛의정원에서는 다음 몬스터가 스폰됩니다.
   - 주니어 그류핀
   - 주니어 셀리온
   - 주니어 라이오너
5. 유저는 몬스터를 사냥해 경험치와 전리품(드롭아이템)을 획득합니다.
6. 전리품(드롭아이템)을 상점에 판매해 메소를 얻습니다.
7. NPC **요정 크리엘**에게서 더 강력한 무기를 구매합니다.
8. 무기를 교체하면 캐릭터 아바타의 장비가 변경되고, 장비 아이템은 소모됩니다.
9. 장착한 무기에 따라 서로 다른 스킬이 사용됩니다.
10. 몬스터를 사냥하다 보면 낮은 확률로 상점 판매 무기를 드롭으로 획득할 수 있습니다.

## 3. 테스트 전 세팅

새로운 월드를 생성한 뒤, 제공된 MOD 파일을 Import합니다.

- `File > Import from file`을 선택하여 테스트 패키지를 불러옵니다.

Import 후 다음 리소스가 Workspace에 제공됩니다.

| 리소스 분류           | 용도                                     | 수정 가능 여부              | 비고                                                                       |
| --------------------- | ---------------------------------------- | --------------------------- | -------------------------------------------------------------------------- |
| Monster Model         | 몬스터 Entity 생성 및 기본 AI 사용       | 수정 금지                   | 기본 AI 수정 금지, 코드에서 컴포넌트 추가 가능(런타임 AddComponent만 가능) |
| NPC Model             | NPC Entity 생성                          | 수정 금지                   | 코드에서 컴포넌트 추가 가능(런타임 AddComponent만 가능)                    |
| Item Image            | 인벤토리/상점/드롭 등 아이템 이미지 표시 | 수정 금지                   | itemId 기준으로 매핑                                                       |
| Equipment Resource    | 인벤토리/상점/드롭 등 아이템 이미지 표시 | 수정 금지                   | 장비 RUID 사용                                                             |
| Skill Effect Resource | 스킬 사용/피격 이펙트 표시               | 수정 금지(RUID 형태로 존재) | UseEffect, Ball, HitEffect 사용                                            |
| UI Resource           | 인벤토리/상점/팝업 UI 구현               | 수정 금지                   | UI 모델이 별도로 없거나 일부 누락된 경우 직접 구성 가능                    |
| Sound Resource        | UI/스킬 사운드 출력                      | 수정 금지(RUID 형태로 존재) | UseSound 및 UI 사운드 사용                                                 |
| Localization Resource | 한국어/영어 표시                         | 수정 금지                   | 다국어 언어 지원                                                           |

> 참고: UI 모델이 명시적으로 제공되지 않은 영역은 의도된 공백일 수 있습니다. 이 경우 지원자는 제공 UI 리소스를 최대한 활용하되, 필요한 UI Entity/Component 구성은 직접 구현할 수 있습니다.
> 참고: UI Resource는 제공된 Sprite/Image 등 원본 리소스를 수정할 수 없다는 의미입니다. UI Model, Component, Layout, Script는 `TheGardenOfLight` 폴더 내부에서 직접 구성할 수 있습니다.
> 참고: LocalWorkSpace를 사용할 경우 AI 도구 활용에 용이해집니다.(https://maplestoryworlds-creators.nexon.com/ko/docs?postId=1165)

### 3.1 작업 폴더 규칙

테스트를 진행하며 작성한 코드, DataSet, Logic, Component, UI, Script 등 모든 신규 파일은 반드시 다음 폴더 내부에 생성해야 합니다.

```text
TheGardenOfLight
```

### 3.2 폴더 내 모든 내용 수정 금지

다음 폴더의 하위 내용은 수정을 금지합니다.

```text
TestPackage
```

`TestPackage` 내부의 모델, 이미지, 사운드, 이펙트 리소스는 원본 그대로 사용해야 합니다.

필요한 컴포넌트는 코드에서 `AddComponent` 방식으로 Entity에 부여하는 것을 원칙으로 합니다.

## 4. 금지 및 제한 사항

다음 사항은 반드시 준수해야 합니다.

| 항목               | 제한                                                                   |
| ------------------ | ---------------------------------------------------------------------- |
| ItemService        | 사용 금지                                                              |
| MultiCast          | 사용 금지                                                              |
| 데이터 저장        | UserDataStorage만 사용 가능                                            |
| TestPackage 리소스 | 수정 금지                                                              |
| Monster AI         | 제공된 모델에 포함된 기본 AI만 사용, 수정 금지                         |
| Client 전용 함수   | Client에서만 실행되는 함수는 가능한 경우 ClientOnly로 분리해야 합니다. |
| Server 전용 함수   | Server에서만 실행되는 함수는 가능한 경우 ServerOnly로 분리해야 합니다. |
| 이벤트 핸들러      | 실행 환경 명시 필수                                                    |

Server 함수는 Client에서 호출될 수 있으므로, Server 함수 내부에서 반드시 보안 검증을 수행해야 합니다.

특히 Server 함수 호출 시 전달되는 파라미터는 클라이언트에서 변조될 수 있다고 가정해야 합니다.

## 5. 데이터 저장 및 불러오기

### 5.1 저장 대상 데이터

다음 데이터는 유저별로 저장되어야 합니다.

- 레벨
- 현재 경험치
- 메소
- 인벤토리 데이터
- 장착 중인 무기

### 5.2 기본 데이터

신규 유저의 기본 데이터는 다음과 같습니다.

| 항목      |                   기본값 |
| --------- | -----------------------: |
| 레벨      |                        1 |
| 경험치    |                        0 |
| 메소      |                      500 |
| 장착 무기 | 없음 또는 기본 무기 상태 |

### 5.3 데이터 로드

유저 데이터 로드는 반드시 `UserEnterEvent`를 받아 처리해야 합니다.

유저가 게임에 접속하면 이전 저장 데이터를 불러와 이어서 플레이할 수 있어야 합니다.

저장 데이터가 없거나 손상된 경우에도 에러 없이 기본 데이터로 초기화되어야 합니다.

로드된 데이터는 서버 기준으로 보정되어야 하며, 특히 다음 값은 유효 범위를 벗어나지 않아야 합니다.

- 레벨
- 경험치
- 메소
- 인벤토리 수량
- 장착 무기 itemId

### 5.4 데이터 저장

유저 데이터 저장은 반드시 `UserLeaveEvent`를 받아 처리해야 합니다.

추가로 다음 상황에서도 저장되어야 합니다.

- 게임 시작 후 5분 단위 자동 저장
- 레벨업 시 즉시 저장

## 6. 레벨 및 경험치

### 6.1 레벨 범위

레벨은 최소 1, 최대 20입니다.

어떤 상황에서도 레벨은 1보다 낮거나 20보다 높아질 수 없습니다.

운영자 커맨드, 데이터 조작, 비정상 요청, 저장 데이터 손상 상황에서도 레벨 범위는 반드시 보정되어야 합니다.

### 6.2 요구 경험치

레벨업에 필요한 경험치는 다음 규칙을 따릅니다.

| 구간     |         요구 경험치 |
| -------- | ------------------: |
| 1 -> 2   |                  50 |
| 2 -> 3   |                 150 |
| 3 -> 4   |                 250 |
| 4 -> 5   |                 350 |
| 이후     | 이전 구간 대비 +100 |
| 19 -> 20 |                9990 |

최대 레벨 20에 도달한 후에는 더 이상 레벨업하지 않습니다.
레벨 20 도달 후 획득 경험치는 증가하지 않으며 현재 경험치는 0으로 유지합니다.
19 -> 20 구간은 예외적으로 9990 경험치를 요구합니다.

### 6.3 UI 동기화

레벨과 경험치는 기본 UI 템플릿을 사용하여 표시해야 합니다.

UI는 서버에서 받은 값을 기준으로 갱신되어야 하며, 클라이언트 임의 값만으로 최종 상태를 판단해서는 안 됩니다.

## 7. 메소

유저는 최초 시작 시 500 메소를 가지고 시작합니다.

몬스터는 절대로 메소를 직접 드롭하지 않습니다.

메소는 다음 경로로만 증가할 수 있습니다.

- 아이템 판매
- 운영자 커맨드
- 기타 서버에서 검증된 보상 처리

메소 값은 인벤토리 UI 내부 영역에 Text Entity를 배치하여 표시해야 합니다.

메소 변경은 서버 기준으로 처리되어야 하며, 클라이언트가 임의로 메소를 증가시킬 수 없어야 합니다.

## 8. 맵 이동

오르비스에서 오른쪽 포탈을 이용하면 빛의정원 맵으로 이동할 수 있어야 합니다.

맵 이동 시 매끄러운 Fade In / Fade Out 처리가 구현되어야 합니다.

맵 이동 로그도 기록되어야 합니다.

## 9. 몬스터 스폰

빛의정원 맵에서는 다음 몬스터가 스폰되어야 합니다.

- 주니어 그류핀
- 주니어 셀리온
- 주니어 라이오너

### 9.1 스폰 영역

몬스터는 제공 이미지의 **노란색 영역 내부**에서만 스폰되어야 합니다.

제공 이미지가 기준이며, 구현 시 스폰 가능한 X/Y 범위를 코드 상수 또는 DataSet으로 명확히 분리해 관리하는 것을 권장합니다.
활동 범위는 해당 AI 특성상 스폰할 경우 문제 없이 돌아가도록 처리되어 있습니다.

> 이미지 자체 수정/추가는 본 문서 반영 범위에서 제외합니다.

### 9.2 스폰 조건

몬스터는 유저가 빛의정원 맵에 있을 때만 스폰되어야 합니다.

스폰 기준은 다음과 같습니다.

| 항목                    |                                  기준 |
| ----------------------- | ------------------------------------: |
| 주니어 그류핀 최대 수   |                                     3 |
| 주니어 셀리온 최대 수   |                                     3 |
| 주니어 라이오너 최대 수 |                                     3 |
| 전체 최대 몬스터 수     |                                     9 |
| 리스폰 대기 시간        |                           사망 후 5초 |
| 맵 정리 대기 시간       | 빛의정원 맵에 유저가 0명이 된 후 30초 |

빛의정원 유저 수가 0명이 되면 신규 리스폰은 중단합니다.
이후 30초 동안 유저가 다시 진입하지 않으면 남아 있는 몬스터를 Destroy합니다.
30초 이내 유저가 재진입하면 정리 타이머를 취소하고 기존 몬스터 상태를 유지하거나 정상 스폰 상태로 복귀합니다.

### 9.3 Entity 재활용 및 Destroy 기준

몬스터 Entity 관리는 다음 기준을 따릅니다.

| 상황                                        | 처리 방식                           |
| ------------------------------------------- | ----------------------------------- |
| 일반적인 리스폰/반복 스폰                   | 가능한 한 Entity 재활용 구조로 구현 |
| 맵에 유저가 없어 일정 시간 후 정리되는 경우 | 남아 있는 몬스터 Entity는 Destroy   |

### 9.4 Monster AI 제한

Monster AI는 제공된 모델에 부착된 기본 AI만 사용해야 하며, 수정할 수 없습니다.

## 10. NPC 배치 및 상점

### 10.1 NPC 배치

서버 인스턴스 시작 시 NPC를 1회 생성해야 하며, 중복 생성되지 않도록 관리해야 합니다.

| NPC         | 위치 기준                 |
| ----------- | ------------------------- |
| 요정 에델   | 제공 이미지의 녹색 범위   |
| 요정 크리엘 | 제공 이미지의 파란색 범위 |

NPC 배치 기준은 제공 이미지이며, 구현 시 NPC 위치 좌표를 코드 상수 또는 DataSet으로 명확히 분리해 관리하는 것을 권장합니다.

NPC에게 `RigidBodyComponent`를 부착하는 것은 금지합니다.

NPC 이름은 `NameTagComponent`를 이용하여 표시해야 합니다.

> 이미지 자체 수정/추가는 본 문서 반영 범위에서 제외합니다.

### 10.2 NPC 상호작용

NPC를 클릭하면 상점 UI가 열려야 합니다.

상점 UI는 반드시 `TestPackage` 내부의 상점 UI 리소스(Sprite)를 사용해야 합니다.

상점에서는 아이템 구매 및 판매가 가능해야 합니다.

상점 판매는 인벤토리에 보유 중인 아이템에 한해 가능하며, 기타 아이템과 장비 아이템 모두 판매할 수 있습니다.
장착 중인 무기는 인벤토리에 존재하지 않으므로 판매할 수 없습니다.
모든 상점 NPC는 판매 기능을 공통으로 제공합니다.

### 10.3 구매/판매 확인 팝업

아이템을 더블클릭하면 즉시 구매/판매되지 않고 확인 팝업이 출력되어야 합니다.

팝업에서 확인 버튼을 눌러야 최종 구매 또는 판매가 처리됩니다.

팝업 문구도 로컬라이징 대상입니다.

### 10.4 UI 사운드

상점 및 인벤토리 UI 상호작용 시 제공된 Sound 리소스를 사용해 사운드가 출력되어야 합니다.

## 14. 스킬 정보

### 14.1 데미지 공식

스킬 데미지는 다음 공식으로 계산합니다.

```text
최소 데미지 = 플레이어 레벨 * 10 + 스킬 공격력 * 3 - 몬스터 방어력
최대 데미지 = 플레이어 레벨 * 10 + 스킬 공격력 * 6 - 몬스터 방어력
```

최종 데미지는 최소 데미지와 최대 데미지 사이의 랜덤 값입니다.

스킬 속성이 몬스터의 약점 속성과 일치하면 최종 데미지를 2배로 적용합니다.
약점이 없음인 몬스터에게는 어떤 속성도 2배 적용되지 않습니다.
스킬 속성이 무인 경우에도 약점 보너스는 적용되지 않습니다.

최종 데미지는 **1보다 작을 수 없습니다.**  
계산 결과가 1 미만인 경우 최종 데미지는 1로 보정합니다.

### 14.2 스킬 범위 표기 기준

스킬 범위는 `width x height` 형식으로 표기합니다.

- `width`: X축 판정 폭
- `height`: Y축 판정 높이
- `바라보는 방향`: 캐릭터가 바라보는 방향으로 X축 범위를 적용
- `캐릭터 중심`: 캐릭터 중심 좌표를 기준으로 X/Y 범위를 적용

## 15. 전투 및 사냥

### 15.1 공격 입력

지원자는 장착 무기에 따라 적절한 스킬이 발동되도록 구현해야 합니다.

각 스킬은 쿨타임, 타격 횟수, 타격 가능 몬스터 수, 범위를 정확히 반영해야 합니다.

공격 입력 키는 구현자가 정할 수 있으나, 구현 설명 문서에 반드시 명시해야 합니다.

### 15.2 플레이어 컨트롤러 기준

플레이어 컨트롤러는 다음 입력과 상태를 서버/클라이언트 역할에 맞게 처리해야 합니다.

| 항목        | 요구사항                                                                                 |
| ----------- | ---------------------------------------------------------------------------------------- |
| 이동        | MSW 기본 이동 동작을 유지하되, 어드민 커맨드에 따른 이동속도/점프력 보정이 반영되어야 함 |
| 공격 입력   | 클라이언트에서 입력을 받고, 서버에서 쿨타임/장착 무기/타격 가능 여부를 검증해야 함       |
| 스킬 연출   | 클라이언트에서 이펙트와 사운드를 재생할 수 있으나, 최종 판정은 서버 기준이어야 함        |
| 장비 상태   | 서버 기준 장착 무기에 따라 사용할 수 있는 스킬이 결정되어야 함                           |
| 플래시 점프 | 어드민 전용 상태가 ON이고 서버 권한 검증을 통과한 경우에만 동작해야 함                   |

### 15.3 충돌 판정

스킬 판정은 반드시 서버 기준으로 검증되어야 합니다.

클라이언트는 공격 입력과 연출을 담당할 수 있으나, 최종 타격 대상과 데미지 반영은 서버에서 검증해야 합니다.

### 15.4 몬스터 사망 처리

몬스터 HP가 0 이하가 되면 사망 처리합니다.

몬스터 사망 처리는 반드시 활성화/비활성화 방식으로 구현해야 합니다.

몬스터 Entity는 다음 생명주기로 순환됩니다.

1. 리스폰(Entity 활성화)
2. 몬스터 HP 0 처리(처치될 경우)
3. 경험치 지급
4. 아이템 드롭
5. 몬스터 킬 로그 기록
6. 사망 상태 전환(Entity 비활성화)
7. 스폰 규칙에 따라 시간이 지나면 다시 1번부터 시작

이미 사망 처리 중이거나 비활성화된 몬스터에 대한 추가 공격 요청은 오류 없이 무시되어야 합니다.

### 15.5 보상 지급 및 드롭 방식

몬스터 처치 보상은 다음 기준으로 지급합니다.

| 보상 종류             | 지급 기준                              |
| --------------------- | -------------------------------------- |
| 경험치                | 마지막 타격자에게 지급                 |
| 일반 드롭/장비 드롭   | 마지막 타격자가 소유권을 지님          |
| 운영자 F8 즉사 커맨드 | 커맨드를 사용한 운영자가 타격자로 처리 |

드롭 처리 방식은 다음 조건을 만족해야 합니다.

1. 몬스터 사망 시 서버에서 드롭 테이블을 기준으로 아이템 드롭 여부를 결정합니다.
2. 드롭 아이템은 바닥에 드롭해야 합니다. (드롭 방식은 RigidBodyComponent 사용 혹은 Tweener을 사용한 자체 구현 모두 상관 없습니다. 드롭 유지 시간은 30초입니다.)
3. 획득 가능 유저와 획득 유효 시간을 서버에서 검증해야 합니다. 또한 드롭된 아이템은 마지막 타격자만 습득할 수 있습니다.
4. 한 몬스터의 드롭 보상은 중복 지급되어서는 안 됩니다.
5. 드롭 테이블의 각 아이템은 독립 확률로 판정합니다. 따라서 한 몬스터에서 여러 종류의 아이템이 동시에 드롭될 수 있습니다.

마지막 타격자는 몬스터 HP를 0 이하로 만든 유효 공격 요청의 요청자입니다.

## 16. 인벤토리

인벤토리는 반드시 `TestPackage` 내부에 제공된 UI 리소스(Sprite)를 사용하여 구현해야 합니다.

인벤토리에는 다음 정보가 표시되어야 합니다.

- 보유 아이템
- 아이템 수량
- 메소
- 장착 중인 무기

아이템 이름은 현재 언어 설정에 맞게 표시되어야 합니다.

인벤토리 UI를 반복해서 열고 닫아도 이벤트가 중복 등록되어서는 안 됩니다.

본 테스트는 싱글 플레이 기준의 기능 구현만으로는 완료로 인정되지 않습니다.

지원자는 최소 2명 이상의 플레이어가 동시에 접속한 상황에서 전투, 보상, UI, 인벤토리, 저장 데이터가 올바르게 동작하는지 검증해야 합니다.

다음 상황에서 데이터, UI, 몬스터 상태, 보상 지급이 일관되게 유지되어야 합니다.

- 여러 플레이어가 같은 몬스터를 동시에 공격하는 상황
- 몬스터가 사망하는 순간 여러 공격 요청이 겹치는 상황
- 한 몬스터에 대한 처치 보상이 중복 지급될 수 있는 상황
- 플레이어별 인벤토리, 경험치, 메소 데이터가 서로 섞일 수 있는 상황
- 다른 플레이어의 사냥/보상/인벤토리 변경이 내 데이터에 영향을 줄 수 있는 상황
- 몬스터 Entity가 Destroy된 후 지연된 공격 또는 보상 로직이 실행될 수 있는 상황
- UI 표시값과 서버 기준 데이터가 서로 달라질 수 있는 상황

### 18.1 멀티 테스트 체크리스트

멀티 테스트 시 최소 다음 항목을 확인해야 합니다.

| 테스트 항목             | 확인 기준                                                                  |
| ----------------------- | -------------------------------------------------------------------------- |
| 동시 접속               | 2명 이상 접속 시 각자의 데이터가 분리되어 표시됨                           |
| 동시 공격               | 같은 몬스터를 동시에 공격해도 HP, 사망 처리, 보상이 1회만 처리됨           |
| 보상 지급               | 마지막 타격자에게 소유하는 의도대로 적용됨                                 |
| 인벤토리 동기화         | 한 플레이어의 아이템 획득/판매가 다른 플레이어 인벤토리에 영향을 주지 않음 |
| UI 동기화               | 서버 값 갱신 후 클라이언트 UI가 올바르게 갱신됨                            |
| 비활성화 이후 지연 요청 | 비활성화된 몬스터에 대한 공격/보상 요청이 오류 없이 무시됨                 |
| 언어별 표시             | 한국어/영어 환경에서 주요 텍스트가 언어 설정에 맞게 출력됨                 |

> 멀티 테스트 관련 이미지 추가는 본 문서 반영 범위에서 제외합니다.

## 19. 보안 및 악용 방지 요구사항

본 과제는 향후 거래 시스템 도입을 고려하여 보안 처리를 중요하게 평가합니다.

Server 함수는 Client에서 호출될 수 있으므로, Server 함수 내부에서는 반드시 보안 검증을 수행해야 합니다.

### 19.1 SenderUserId 활용

Server 함수에서는 `SenderUserId`를 활용하여 요청자를 검증해야 합니다.

클라이언트가 전달한 `userId`, `targetUserId`, `itemId`, `count`, `meso`, `level`, `exp`, `damage` 등의 값은 신뢰해서는 안 됩니다.

## 20. 로컬라이징

본 과제는 한국어와 영어를 지원하는 기본 로컬라이징 구조를 요구합니다.

아이템 이름, 몬스터 이름, NPC 이름, UI 문구, Toast 문구, 상점 팝업 문구, 운영자 관련 문구는 표시 문자열을 코드에 직접 하드코딩하지 않고, 고정 `textKey`를 기준으로 언어별 텍스트를 조회하는 방식으로 구현해야 합니다.

### 20.1 지원 언어

| 언어   | 코드 |
| ------ | ---- |
| 한국어 | `ko` |
| 영어   | `en` |

기본 언어는 한국어입니다.

### 20.2 로컬라이징 대상

다음 텍스트는 한국어와 영어를 모두 지원해야 합니다.

- 아이템 이름
- 몬스터 이름
- NPC 이름
- 인벤토리 UI 문구
- 상점 UI 문구
- 구매/판매 확인 팝업 문구
- 레벨/경험치/메소 UI 문구
- Toast 문구
- 운영자 기능 관련 문구

### 20.3 LocalizationService 사용

`_LocalizationService`를 적극 활용하여 한국어 환경과 영어 환경에서 모두 원활하게 플레이할 수 있도록 구현해야 합니다.

Scene 옆 Play 또는 Preview 탭에서 테스트 플레이 언어를 변경하여, 모든 주요 텍스트가 현재 언어에 맞게 출력되는지 검증해야 합니다.

### 20.4 저장 데이터와 로컬라이징 분리

저장 데이터와 서버 로직은 언어와 무관한 고정 ID를 기준으로 처리되어야 합니다.

예시:

```text
1372005 = Wooden Wand / 우드 완드
```

한국어 환경에서 획득한 우드 완드와 영어 환경에서 표시되는 Wooden Wand는 같은 `itemId`를 가진 동일 아이템으로 처리되어야 합니다.

### 20.5 로컬라이징 리소스 사양 요약

로컬라이징 리소스는 다음 기준으로 구성합니다.

| 필드     | 설명                      | 예시                                             |
| -------- | ------------------------- | ------------------------------------------------ |
| textKey  | 코드에서 참조하는 고정 키 | `item.1372005.name`                              |
| ko       | 한국어 표시 문자열        | `우드 완드`                                      |
| en       | 영어 표시 문자열          | `Wooden Wand`                                    |
| category | 분류                      | `item`, `monster`, `npc`, `ui`, `toast`, `admin` |

권장 textKey 예시는 다음과 같습니다.

| 대상        | textKey 예시                                    |
| ----------- | ----------------------------------------------- |
| 아이템 이름 | `item.{itemId}.name`                            |
| 몬스터 이름 | `monster.{monsterId}.name`                      |
| NPC 이름    | `npc.{npcId}.name`                              |
| UI 문구     | `ui.inventory.title`, `ui.shop.buyConfirm`      |
| Toast 문구  | `toast.itemFull`, `toast.notEnoughMeso`         |
| 어드민 문구 | `admin.command.success`, `admin.command.denied` |

## 21. 운영자 기능

본 과제에는 라이브 서비스 운영 상황을 가정한 운영자 전용 기능 구현이 포함됩니다.

운영자 기능은 일반 유저가 접근할 수 없어야 하며, 반드시 서버 기준 권한 검증을 거쳐 실행되어야 합니다.

클라이언트 UI 표시 여부만으로 권한을 판단해서는 안 됩니다.

### 21.1 어드민 판별

어드민은 `AdminDataSet`을 생성하여 관리합니다.

`AdminDataSet`의 CSV에 포함된 `UserId`가 접속할 경우, 해당 유저는 Admin으로 게임을 시작합니다.

어드민 여부는 서버에서 판별해야 합니다.

클라이언트에서 임의로 어드민 상태를 변경할 수 없어야 합니다.

일반 유저가 어드민 커맨드 이벤트를 직접 호출해도 기능이 실행되어서는 안 됩니다.

### 21.2 어드민 커맨드

테스트 시간 내 어드민 전용 패널 UI 구현은 필수가 아닙니다.

대신 F1~F12 Key를 이용하여 어드민 커맨드를 사용할 수 있어야 합니다.

| Key | 기능                                       | 단위/처리 기준                    |
| --- | ------------------------------------------ | --------------------------------- |
| F1  | 이동속도 증가                              | `WalkSpeed +0.1`                  |
| F2  | 이동속도 감소                              | `WalkSpeed -0.1`                  |
| F3  | 점프력 증가                                | `WalkJump +0.05`                  |
| F4  | 점프력 감소                                | `WalkJump -0.05`                  |
| F5  | 자신의 메소 증가                           | `+30000`                          |
| F6  | 자신의 메소를 0으로 설정                   | 현재 메소를 즉시 0으로 변경       |
| F7  | 자신의 데이터 초기화 후 게임 이탈          | 빠르게 3번 입력 시 발동           |
| F8  | 현재 맵의 모든 몬스터에게 강제 데미지 적용 | 각 몬스터에게 `99999` 데미지 적용 |
| F9  | 레벨 증가                                  | `+1`, 최대 20 보정                |
| F10 | 경험치 증가                                | `+100`, 레벨업 규칙 적용          |
| F11 | 어드민 전용 플래시 점프 ON                 | 상태 ON                           |
| F12 | 어드민 전용 플래시 점프 OFF                | 상태 OFF                          |

F6은 이후 메소 획득을 영구 차단하는 기능이 아니라, 커맨드 실행 시점의 자신의 메소를 0으로 변경하는 기능입니다.
F7 발동 시 서버 기준 데이터를 기본값으로 초기화하고 즉시 저장한 뒤, 가능한 경우 유저를 월드에서 이탈 처리합니다.(UserService의 Kick 기능 사용 권장)
F8 강제 데미지는 스킬 데미지 공식, 방어력, 약점 배율을 적용하지 않는 고정 데미지입니다.

### 21.3 이동속도 변경

`RigidBodyComponent`의 `WalkSpeed` 값을 조정합니다.

| 항목           |  값 |
| -------------- | --: |
| 증가/감소 단위 | 0.1 |
| 최소값         | 0.8 |
| 최대값         | 1.4 |

### 21.4 점프력 변경

`RigidBodyComponent`의 `WalkJump` 값을 조정합니다.

| 항목           |   값 |
| -------------- | ---: |
| 증가/감소 단위 | 0.05 |
| 최소값         | 0.75 |
| 최대값         | 1.25 |

### 21.5 데이터 초기화 커맨드

F7 커맨드는 빠르게 3번 눌러야 발동해야 합니다.

F7 커맨드가 발동되면 자신의 데이터를 초기화하고 게임에서 이탈합니다.

- 첫 번째 F7 입력 시점부터 2초 타이머를 시작합니다.
- 2초 이내 3회 입력되면 데이터 초기화가 실행됩니다.
- 2초가 지나면 입력 횟수는 초기화됩니다.
- 발동 시 서버에서 Admin 권한을 재검증합니다.

초기화 대상:

- 레벨
- 경험치
- 메소
- 인벤토리
- 장착 무기
  즉, 플레이어의 모든 데이터

### 21.6 몬스터 즉사 커맨드

F8 커맨드는 현재 맵에 존재하는 모든 몬스터에게 99999 데미지를 강제로 적용합니다.
비활성화 상태, 사망 처리 중, 이미 보상 지급이 완료된 몬스터는 대상에서 제외하거나 오류 없이 무시해야 합니다.

이때 커맨드를 사용한 Admin이 타격자로 처리되어야 하며, 보상도 해당 Admin 기준으로 지급되어야 합니다.

다만 몬스터 사망 처리와 보상 지급은 여전히 몬스터별로 단 한 번만 수행되어야 합니다.

### 21.7 어드민 전용 플래시 점프

F11을 누르면 어드민 전용 플래시 점프가 ON 됩니다.

F12를 누르면 OFF 됩니다.

플래시 점프가 ON 된 상태에서는 점프 중 다시 점프키를 눌렀을 때 추가 점프가 가능해야 합니다.

세부 사양은 다음 조건을 따릅니다.

| 항목        | 요구사항                                     |
| ----------- | -------------------------------------------- |
| 사용 권한   | Admin만 사용 가능                            |
| 서버 검증   | 일반 유저가 이벤트를 직접 호출해도 실행 금지 |
| 발동 조건   | 공중 상태에서 점프 입력 시 발동              |
| 사용 제한   | 지면에 닿기 전 최대 1회 발동                 |
| 쿨타임      | 최소 0.3초 이상 적용                         |
| OFF 상태    | 발동하지 않음                                |
| 사용 이펙트 | c339dabca3f340b09d1be4d7ddc65270             |

플래시 점프의 이동 거리와 방향 구현은 자유입니다.
단, Admin 권한, ON/OFF 상태, 공중 상태, 지면 착지 전 1회 제한, 최소 0.3초 쿨타임은 서버 기준으로 검증해야 합니다.

## 22. 운영 로그

라이브 서비스 운영 이슈 대응을 위해 로그 기능을 구현해야 합니다.

로그 기능은 Google Spreadsheet, Apps Script, MSW 내부 HttpService를 이용하여 구현합니다.

또한 기능을 연결한 운영 로그가 실제로 기록되는 Google Spreadsheet URL은 구현된 내용이 담겨있는 문서에 포함되어야 합니다.

로그 전송 실패는 게임 행동 처리 결과를 롤백하지 않습니다.
단, 로그 전송 실패가 발생해도 게임 플레이가 중단되거나 에러로 종료되어서는 안 되며, 가능하다면 재시도 또는 내부 경고 로그를 남기는 구조를 권장합니다.

### 22.1 로그 기록 대상

다음 유저 행동은 반드시 기록되어야 합니다.

- 아이템 구매 로그
- 아이템 판매 로그
- 메소 변화 로그
- 아이템 획득 로그
- 레벨업 로그
- 몬스터 킬 로그
- 맵 이동 로그
- 데이터 저장 로그
- 어드민 커맨드 실행 로그
- 보안 검증 실패 로그

### 22.2 actionId 기준

운영 로그에는 현지화된 문자열만 남기지 말고, 분석 가능한 고정 `actionId`를 함께 남겨야 합니다.

`actionId`는 로그 분석, 장애 대응, 운영툴 필터링에 사용하는 고정 행동 ID입니다. 언어 설정이나 UI 문구와 무관하게 항상 동일한 값을 사용해야 합니다.

권장 `actionId` 예시는 다음과 같습니다.

| 분류           | actionId 예시    |
| -------------- | ---------------- |
| 아이템 구매    | `BuyItem`        |
| 아이템 판매    | `SellItem`       |
| 메소 변경      | `ChangeMeso`     |
| 아이템 획득    | `AcquireItem`    |
| 레벨업         | `LevelUp`        |
| 몬스터 킬      | `KillMonster`    |
| 맵 이동        | `ChangeMap`      |
| 데이터 저장    | `SaveUserData`   |
| 어드민 커맨드  | `AdminCommand`   |
| 보안 검증 실패 | `SecurityDenied` |

### 22.3 로그 필드

로그에는 가능한 한 다음 정보를 포함해야 합니다.

| 필드         | 설명                          | 기본값/없을 때 처리                      |
| ------------ | ----------------------------- | ---------------------------------------- |
| timestamp    | 실행 시간                     | 서버 시간 기준 필수                      |
| actionId     | 행동 ID                       | 필수                                     |
| userId       | 실행 유저                     | 필수                                     |
| targetUserId | 대상 유저, 필요 시            | 없으면 빈 문자열 또는 `nil`              |
| mapName      | 맵 이름, 필요 시              | 없으면 빈 문자열 또는 `nil`              |
| itemId       | 아이템 ID, 필요 시            | 없으면 빈 문자열 또는 `nil`              |
| monsterId    | 몬스터 ID, 필요 시            | 없으면 빈 문자열 또는 `nil`              |
| amount       | 수량 또는 변경값              | 없으면 `0`                               |
| beforeValue  | 변경 전 값                    | 없으면 빈 문자열 또는 `nil`              |
| afterValue   | 변경 후 값                    | 없으면 빈 문자열 또는 `nil`              |
| result       | 로그 대상 행동의 처리 결과    | `Success`, `Fail`, `Denied` 중 하나 권장 |
| reason       | 실패, 거부, 예외 발생 시 사유 | 성공 시 빈 문자열 또는 `nil`             |

`result`와 `reason`은 로그 전송 성공 여부가 아니라, **로그로 남기는 행동 자체의 처리 결과와 사유**를 의미합니다.

예시:

- 아이템 구매 성공: `result=Success`, `reason=nil`
- 메소 부족으로 구매 실패: `result=Fail`, `reason=NotEnoughMeso`
- 일반 유저의 어드민 커맨드 호출 거부: `result=Denied`, `reason=NotAdmin`

### 22.4 로그 예시

```text
[UserAction] actionId=BuyItem userId=123 itemId=1372005 amount=1 result=Success
[UserAction] actionId=SellItem userId=123 itemId=4000073 amount=10 result=Success
[AdminAction] actionId=AdminCommand userId=123 amount=30000 result=Success reason=F5_AddMeso
[Security] actionId=SecurityDenied userId=999 result=Denied reason=NotAdmin
```

## 23. 제출물

지원자는 기간 내 다음 결과물을 제출해야 합니다.

| 제출물                 | 설명                                                                  |
| ---------------------- | --------------------------------------------------------------------- |
| MOD 파일               | 구현 완료 또는 미완성 상태의 월드 파일                                |
| Google Spreadsheet URL | 운영 로그가 기록되는 시트 URL                                         |
| 구현 설명 문서         | 구현 범위, 미구현 범위, 테스트 방법, 조작 키, 어드민 테스트 방법 작성 |
| 조작 방법 문서         | 공격, 인벤토리, 상점, 어드민 커맨드, 테스트 계정/권한 설정 방법       |
| 멀티 테스트 결과       | 최소 2인 접속 테스트 결과 요약                                        |
| 미완성 사유            | 미완성 제출 시 반드시 포함                                            |

제출 기간 내에는 완성도가 부족하더라도 반드시 제출해야 합니다. 제출하지 않은 경우보다, 미완성 범위와 원인을 명확히 정리한 제출물을 더 높게 평가할 수 있습니다.
